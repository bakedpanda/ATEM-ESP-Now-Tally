# Plan: Web-based Firmware Flashing

Flash ESP32-C3 boards over USB directly from the dashboard. WiFi credentials and
server address are entered in the UI before flashing — the firmware binary itself
contains no credentials and can be committed to the repo.

---

## How it works

Flashing happens entirely in the browser via the **Web Serial API** (Chrome/Edge).
The server is not involved in the USB communication at all, so Docker or any other
deployment method is unaffected.

Two binaries are flashed in one operation:

- `firmware.bin` — compiled once, credential-free, committed to the repo
- `nvs.bin` — generated by the server on demand from the credentials the user
  entered, contains WiFi SSID/password and server host/port

---

## Step 1 — Firmware: read credentials from NVS

Remove the `WIFI_SSID`, `WIFI_PASSWORD`, `SERVER_HOST`, and `SERVER_PORT` defines
from `firmware/include/config.h`. Replace with Preferences reads in the two files
that use them.

**`firmware/src/wifi_manager.cpp`** — replace `WiFi.begin(WIFI_SSID, WIFI_PASSWORD)`:

```cpp
#include <Preferences.h>

void wifiManagerInit() {
    Preferences prefs;
    char ssid[64] = {};
    char pass[64] = {};
    prefs.begin("tally", true);
    prefs.getString("ssid", ssid, sizeof(ssid));
    prefs.getString("pass", pass, sizeof(pass));
    prefs.end();
    WiFi.mode(WIFI_AP_STA);
    WiFi.begin(ssid, pass);
    lastAttempt = millis();
}
```

**`firmware/src/ws_client.cpp`** — replace `ws.begin(SERVER_HOST, SERVER_PORT, "/bridge")`:

```cpp
#include <Preferences.h>

void wsClientInit(BridgeState* state, RoleCallback onRole, IdentifyCallback onIdentify) {
    // ... existing state setup ...
    Preferences prefs;
    char host[64] = {};
    prefs.begin("tally", true);
    prefs.getString("srv_host", host, sizeof(host));
    uint16_t port = prefs.getUShort("srv_port", 8259);
    prefs.end();
    ws.begin(host, port, "/bridge");
    ws.onEvent(wsEvent);
    ws.setReconnectInterval(3000);
}
```

**NVS keys** (namespace `"tally"`):

| Key | Type | Contains |
|-----|------|----------|
| `ssid` | string | WiFi network name |
| `pass` | string | WiFi password |
| `srv_host` | string | Server IP or hostname |
| `srv_port` | uint16 | Server port (default 8259) |

## Step 2 — Firmware: handle unconfigured state

If the device boots with empty NVS (i.e. flashed without an `nvs.bin`) it will
fail to connect. Add a timeout in `main.cpp` setup — if WiFi hasn't connected
within 15 seconds, switch to a **fast red flash** LED state and halt rather than
hanging silently. This makes it obvious the device needs to be re-flashed with
credentials.

---

## Step 3 — Firmware: build and distribute binaries

Add a `firmware/dist/` directory (git-tracked). After each build, copy the four
files that esptool-js needs into it:

| File | Flash offset |
|------|-------------|
| `bootloader.bin` | `0x0000` |
| `partitions.bin` | `0x8000` |
| `boot_app0.bin` | `0xe000` |
| `firmware.bin` | `0x10000` |

Add a PlatformIO post-build extra script (`firmware/copy_dist.py`) that copies
these from `.pio/build/tally/` automatically after every `pio run`.

The fifth binary — `nvs.bin` at offset `0x9000` — is generated by the server and
never stored on disk.

## Step 4 — Server: NVS partition generator

Add **`server/src/nvs.js`** — a self-contained utility that encodes a small set of
key-value pairs into a valid ESP32 NVS partition binary (Espressif's documented
format: 4 KB pages with a header, entry bitmap, and packed entries).

```js
// Rough API
export function buildNvsPartition({ ssid, pass, srvHost, srvPort }) {
  // Returns a Buffer of exactly 0x4000 (16 KB) bytes
}
```

No npm dependencies. The NVS format for a handful of string and integer keys is
straightforward to implement in ~100 lines.

---

## Step 5 — Server: network profiles

Add a `networks` array to `config.json` (extend `DEFAULT_CONFIG` in `config.js`):

```json
"networks": [
  { "name": "Studio", "ssid": "StudioWiFi", "password": "secret" }
]
```

Passwords are stored in plaintext — acceptable for a local-network appliance.

---

## Step 6 — Server: new routes

Add to `routes.js`:

```
GET  /flash                   → flash.html
GET  /api/networks            → list saved profiles
POST /api/networks            → save / update a profile  { name, ssid, password }
DEL  /api/networks/:name      → delete a profile
GET  /firmware/:file          → serve static files from firmware/dist/
GET  /firmware/nvs.bin        → generate NVS binary on demand
                                query: ?ssid=&pass=&host=&port=
```

The `/firmware/nvs.bin` endpoint validates that `ssid` and `host` are non-empty,
calls `buildNvsPartition()`, and returns the result as
`Content-Type: application/octet-stream`.

---

## Step 7 — Server: flash page

New **`server/views/flash.html`** with three sections:

**Network**
- Dropdown of saved profiles — selecting one populates the fields
- SSID input + password input (masked, with show/hide toggle)
- "Save profile" button — POSTs to `/api/networks`

**Server address**
- Host field pre-populated from `window.location.hostname`
- Port field pre-populated from `window.location.port`

**Flash**
- "Connect & Flash" button — uses **esptool-js** directly via Web Serial
  (loaded from CDN or bundled; no esp-web-tools wrapper needed)
- On click:
  1. Validate that SSID and host are not empty
  2. Open Web Serial port picker
  3. Fetch `firmware/bootloader.bin`, `firmware/partitions.bin`,
     `firmware/boot_app0.bin`, `firmware/firmware.bin`, and
     `firmware/nvs.bin?ssid=…&pass=…&host=…&port=…` in parallel
  4. Flash all five files to their respective offsets with progress shown in a
     log panel below the button
- Error display for empty credentials, serial connection failure, or flash errors

## Step 8 — Nav update

Add `<a href="/flash">Flash</a>` to the nav bar in all four views:
`dashboard.html`, `assign.html`, `settings.html`, `flash.html`.

## What a first-time setup looks like

1. Build firmware with PlatformIO — binaries land in `firmware/dist/` via the
   post-build script
2. Open `http://localhost:8259/flash`
3. Enter WiFi credentials, save as a profile
4. Confirm server host/port (defaults are correct for localhost)
5. Plug in the ESP32-C3 via USB, click "Connect & Flash", pick the serial port
6. Browser flashes all five partitions (~30 s), device reboots, connects to WiFi,
   appears in the dashboard

Subsequent boards on the same network: pick the saved profile, plug in, flash.
No credentials to re-enter.
